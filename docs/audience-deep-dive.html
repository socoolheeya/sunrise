1. Audience 기능과 역할

Audience는 마케터가 "지금 공략할 고객 집단"을 정의·산출하도록 돕는 영역입니다. 바로 쓰는 표준 템플릿 카탈로그를 제공하고, 조건 DSL로 모수를 실시간 미리보기(preview)하며, 확정된 대상자를 캠페인이 참조할 read model로 저장(materialize)합니다.

기능역할(무엇을 하나)핵심 산출
템플릿 카탈로그이커머스 표준 오디언스를 바로 사용58개 템플릿(11개 카테고리), category/query 필터
템플릿 상세단일 템플릿의 조건 계약 확인rule(DSL) + 추천 채널/trigger/tag + evaluable
preview(미리보기)조건 변경 시 예상 모수 즉시 확인matched_count · sample · unsupported · rule_hash
materialize(확정)대상자 집합을 캠페인 연결용으로 고정audience_id read model(member_count·rule_hash)
조건 DSL이벤트·프로필·예측 score를 트리로 조합all/any × event_count/profile/score
예측 score 조건구매가능성 등 ML 점수로 타게팅purchase_score·churn_risk·affinity 조건

카테고리(11) — acquisition · browse · cart · channel · churn · experiment · loyalty · prediction · purchase · retention · value. 설계 — 모든 조회 tenant 격리, preview는 단기 TTL 캐시, materialize는 멱등 upsert, 미지원 조건은 명시적으로 노출(빈 모수 silent 금지).

3. 조건 DSL

오디언스 조건은 중첩 가능한 규칙 트리입니다. all(AND)/any(OR) 아래에 3종 조건을 조합합니다.

# 트리 구조 { "all": [ 조건 | 하위트리, ... ] } # 모두 만족(AND) { "any": [ 조건 | 하위트리, ... ] } # 하나라도(OR) # 조건 타입 3종 { "type":"event_count", "event":"cart_add", "window_days":7, "op":"gte", "value":1 } { "type":"profile", "field":"days_since_last_purchase", "op":"gte", "value":30 } { "type":"score", "name":"purchase_score", "op":"gte", "value":0.6 } # 연산자(op) eq · in · gte · gt · lte · lt # in 은 집합 포함 # event_count: window_days 지정 시 [end − window_days, end] 하위 윈도우만 카운트 # rule_hash = sha256(canonical_json(rule)) → preview 캐시 키 · materialize 식별
조건 타입평가 대상예시
event_count방문자의 특정 이벤트 횟수(window_days 윈도우)최근 7일 cart_add ≥ 1
profile방문자 집계/파생 필드(구매수·매출·최근성·상품가치 등)days_since_last_purchase ≥ 30
score예측 모델 아티팩트 점수purchase_score ≥ 0.6

4. profile feature 백필 (3계층)

preview는 기간 이벤트와 상품 feature를 방문자별로 집계해 profile 필드를 만듭니다. 데이터 출처에 따라 3계층으로 나뉘며, 외부 소스가 필요한 필드는 평가 불가로 명시됩니다.

5. 예측 score 조건

score 조건은 예측 모델 아티팩트(prediction_model.json, 모델 레지스트리로 버전 관리)를 로드해 방문자 점수를 산출합니다. Audience와 Prediction이 동일 아티팩트를 공유해 일관성을 유지합니다.

지원 score: purchase_score · churn_risk · category_affinity · product_affinity · next_best_offer # 방문자 프로필 → VisitorPredictionFeatures → MultiHeadLogisticPredictionModel purchase_score / churn_risk = sigmoid(bias + Σ wᵢ·fᵢ) # 예측 모듈과 동일 head

13. 용어집

오디언스 템플릿 (Audience Template)

바로 사용할 수 있는 표준 고객 세그먼트 정의. template_id·name·category·rule(DSL)·추천 채널·추천 trigger·tags로 구성. 카탈로그 버전은 audience-template-catalog.v1, 현재 58개.

조건 DSL · 규칙 트리 (rule tree)

오디언스 조건을 표현하는 중첩 구조. all(AND)/any(OR) 노드 아래에 조건을 둡니다. 단순 enum 필터가 아니라 방문 빈도·구매 이력·상품 관심·ML 점수를 자유 조합할 수 있습니다.

조건 타입: event_count / profile / score

event_count: 특정 이벤트 횟수(window_days 윈도우). profile: 방문자 집계/파생 필드. score: 예측 모델 점수. 각 조건은 op(eq·in·gte·gt·lte·lt)와 value로 비교합니다.

window_days

event_count 조건의 시간 창. 지정 시 [end − window_days, end] 하위 윈도우 내 이벤트만 셉니다(예: "24시간 이탈" vs "7일 이탈" 구분). API 윈도우(start/end)와 별개로 조건 단위로 적용.

preview (미리보기) · matched_count · sample_visitor_ids

조건을 실데이터로 평가해 예상 모수를 즉시 반환. matched_count(매칭 인원), sample_visitor_ids(샘플 ID), unsupported_conditions를 포함. 단기 TTL 캐시(키에 rule_hash).

materialize (확정) · audience_id · member_count

평가 결과를 read model(audience_materializations)로 저장해 캠페인이 참조. audience_id로 식별, member_count는 확정 시점 모수. 같은 (tenant, audience_id)는 멱등 upsert.

rule_hash

조건 DSL을 정규화(canonical JSON)해 만든 sha256 해시. 동일 조건을 식별하는 키로, preview 캐시 키와 materialize 행에 저장됩니다.

unsupported_conditions

이 배포에서 평가할 수 없는 조건 목록(예: profile:kakao_opt_in). 해당 조건은 False로 처리하되 결과에 명시해, "빈 모수"의 원인을 조용히 숨기지 않습니다.

evaluable · external_fields (카탈로그 메타)

템플릿이 외부 소스 전용 필드를 참조하는지 여부. external_fields가 비어 있으면 evaluable=true(현 배포에서 실모수 평가 가능). 콘솔이 "평가 불가" 템플릿을 구분하는 데 사용.

profile feature 백필 (3계층)

profile 필드의 출처 분류 — ① 이벤트 도출, ② 상품 feature join, ③ 외부 소스(consent/쿠폰/배송). ①②는 지원, ③은 미지원으로 명시. 백필 확대로 평가 가능 템플릿이 늘어납니다(현재 49/58).

예측 score 조건

구매가능성(purchase_score)·이탈위험(churn_risk)·상품반응(affinity) 등 ML 점수를 오디언스 조건으로 사용. Prediction과 동일한 모델 아티팩트를 공유해 점수 일관성을 보장합니다.

category / query 필터

템플릿 목록 필터. category는 정확 일치, query는 name/description/tags에 대한 부분 일치(casefold). 11개 카테고리(acquisition·browse·cart·channel·churn·experiment·loyalty·prediction·purchase·retention·value).

as_of · status

as_of: 오디언스를 평가한 기준 시각(evaluated_at). status: 현재 materialize 시 active로 저장(향후 scheduled refresh/만료 확장 여지).

read model (참고)

조회/소비에 최적화해 미리 만들어 둔 데이터. audience_materializations는 캠페인이 대상자 집합을 빠르게 참조하기 위한 audience read model입니다. (자세한 정의는 analytics 용어집 참조.)

Port / Adapter (참고)

Audience는 단순 카탈로그+평가라 별도 Repository 포트 없이, 유스케이스가 AsyncSession으로 직접 events/product_features를 읽고 audience_materializations에 씁니다. 예측 점수만 Prediction 아티팩트(레지스트리)에 의존.
