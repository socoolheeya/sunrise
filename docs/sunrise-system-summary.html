전체 아키텍처

계층 구조 (Clean Architecture)

모든 모듈은 domain → application → adapters 의존성 규칙을 따릅니다. 도메인은 외부 프레임워크를 모르고, 어댑터(HTTP·SQL·ClickHouse·Kafka)만 인프라를 참조합니다.

flowchart LR subgraph A["adapters"] HTTP["HTTP Router (FastAPI)"] SQL["SQL Repository"] CHR["ClickHouse Repository"] end subgraph AP["application"] UC["Query / Refresh Use Cases"] end subgraph D["domain"] MODEL["Entity · VO · 순수 계산 함수"] PORT["Repository Port (ABC)"] end HTTP --> UC UC --> PORT UC --> MODEL SQL -. implements .-> PORT CHR -. implements .-> PORT

포트(AnalyticsRepository 등 ABC)는 SQL과 ClickHouse 두 어댑터가 구현하므로, 백엔드 전환이 HTTP/유스케이스 계약을 바꾸지 않습니다.

Read Model 머티리얼라이즈 패턴 Read Model

주문 단위 중복제거·세그먼트 이동·코호트 리텐션은 매 조회마다 raw scan하지 않고, refresh(배치 잡)로 목적별 read model을 적재한 뒤 조회합니다. 운영(ClickHouse) 모드는 스트리밍 MV가 동일 역할을 합니다.

sequenceDiagram participant Job as Scheduler / Operator participant API as FastAPI participant UC as Refresh UseCase participant Repo as SQL Repository participant T as Read Model Table Job->>API: POST /…/refresh (tenant, window) API->>UC: execute(tenant, start, end) UC->>Repo: raw events 조회 Repo-->>UC: events UC->>UC: 도메인 fold (dedup/분류/버킷) UC->>T: upsert (또는 delete-then-insert) API->>API: 해당 read 캐시 키 무효화 Note over T: 이후 조회는 read model 을 직접 읽음

적용 대상: order_facts, customer_segment_daily, cohort_retention. 각 refresh는 멱등(upsert/replace)이며, refresh 직후 관련 Redis 캐시 키를 삭제해 stale 응답을 방지합니다.

Prediction 예측 ML

학습된 멀티헤드 로지스틱 아티팩트로 구매가능성·이탈위험·상품반응을 스코어링하고, CLV는 BG/NBD + Gamma-Gamma 확률 모델로 산출합니다. 아티팩트는 holdout backtest를 포함한 실제 학습 산출물이며 모델 레지스트리로 버전 관리됩니다.

flowchart LR REQ["visitor_ids"] --> FEAT["feature 추출 (repository)"] FEAT --> XF["feature 변환
cap_log · recency"] XF --> M{"model head"} M -->|purchase_score| PS["sigmoid"] M -->|churn_risk| CR["sigmoid → retarget days"] M -->|product_affinity| PA["sigmoid → 정렬"] REQ --> POP["population_features
(전체 구매 고객)"] POP --> FIT["BG/NBD + Gamma-Gamma
MLE 적합"] FIT --> CLV["CLV: 생존확률·기대구매·예측CLV"] M --> EXP["explain : fᵢ·wᵢ 기여도"]

공식 — feature 변환 & 로지스틱

cap_log(x, cap) = clamp01( ln(1+max(x,0)) / ln(1+cap) ) recency_signal = clamp01(1 − min(recency,90)/90) inactivity_signal= clamp01(min(days_since_seen,90)/90) # 헤드별 로지스틱 logit = bias_head + Σ wᵢ·fᵢ score = sigmoid(logit) = 1 / (1 + e^(−logit)) # churn → 재타게팅 시점 retarget_days = 1 (risk≥0.7) | 3 (≥0.35) | 7 (else) explain : contributionᵢ = fᵢ · wᵢ # 선형 기여도 (top reasons)

공식 — CLV (BG/NBD + Gamma-Gamma)

# 입력: frequency x(반복구매), recency t_x, T(첫구매~관측), monetary m̄ # BG/NBD 모수 (r, α, a, b) 는 모집단 MLE(Nelder-Mead)로 적합 P(alive) = 1 / (1 + δ_{x>0}·(a/(b+x−1))·((α+T)/(α+t_x))^(r+x)) E[Y(t)|x,t_x,T] = (a+b+x−1)/(a−1) · (1 − ((α+T)/(α+T+t))^(r+x)·₂F₁(…)) / denom # Gamma-Gamma posterior mean (모집단 평균으로 수축) E[M | x, m̄] = (M_pop·k + x·m̄) / (x + k) predicted_clv = E[Y(horizon)] · E[M] # 표본 부족·콜드스타트 시 보수적 휴리스틱으로 graceful fallback

공식 — feature 변환 보강 / backtest

# 학습 artifact: in-sample metrics 와 별도로 holdout backtest 산출 metrics = in-sample AUC/log_loss (전체 학습본) backtest = holdout(매 5번째) out-of-sample AUC/log_loss # 과적합 검증 drift_baseline = 학습 feature 분포(서빙 drift 비교 기준) # 로더는 hand-filled/스키마 불일치 artifact 를 거부(서빙==학습 산출물 보장)
CLV가 확률 모델(BG/NBD·Gamma-Gamma, 순수 stdlib MLE)로 구현됐습니다. drift는 baseline 산출·노출까지(실측 PSI 비교는 후속). explain은 로지스틱 선형 기여도(SHAP 아님).
