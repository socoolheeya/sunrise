Sunrise / Sunrise Core 시스템 요약

이 문서는 현재 개발된 sunrise Python/FastAPI 분석·수집 서비스와 sunrise-core Spring Kotlin 운영 코어 서비스를 기준으로 아키텍처, 기능별 시퀀스, 실제 연동 방법을 정리한 실행 문서입니다.

수집 경로FastAPI → Kafka → ClickHouse
분석/Feature 저장소ClickHouse 물리 DB
운영 코어Spring Kotlin + PostgreSQL
추천 랭킹ML v3 + 상품 가치 feature

1. 시스템 역할 분리

프로젝트 역할 주요 모듈 저장소/인프라
sunrise 고객 서비스에서 발생한 행동 이벤트를 수집하고, 분석·예측·추천·AI 제안을 제공한다. ingestion, analytics, prediction, recommendation, ai, events, core Kafka/Redpanda, ClickHouse, Redis, PostgreSQL
sunrise-core 테넌트, API key, 캠페인 상태 전이, 감사 로그, 도메인 이벤트 outbox를 관리한다. tenant, campaign, shared PostgreSQL, Flyway, JPA

실운영에서는 sunrise-core가 고객사/테넌트와 운영 리소스를 관리하고, sunrise가 고속 이벤트 수집 및 OLAP 조회를 담당하는 구조가 자연스럽습니다. 현재 두 프로젝트는 물리적으로 분리되어 있으며, API key 발급 결과를 sunrise의 `SUNRISE_API_KEYS_JSON` 또는 별도 secret/IAM 저장소에 동기화하는 구성이 필요합니다.

2. 아키텍처 구성도

클라이언트/운영자 API/애플리케이션 스트리밍 OLAP Core 운영 DB

4.1 실운영 이벤트 수집

sequenceDiagram
    autonumber
    participant Client as 고객 웹/앱/백엔드
    participant API as sunrise FastAPI
    participant Auth as Tenant Resolver
    participant Kafka as Redpanda raw.events
    participant CHQ as ClickHouse raw_events_queue
    participant MV as Materialized Views
    participant CH as ClickHouse read models

    Client->>API: POST /v1/collect + X-Sunrise-Key
    API->>Auth: API key로 tenant_id 확인
    Auth-->>API: tenant-a
    API->>API: schema_version, event type, size, rate limit 검증
    API->>Kafka: JSONEachRow event publish
    Kafka-->>API: ack=all
    API-->>Client: 202 accepted
    Kafka-->>CHQ: Kafka Engine consume
    CHQ-->>MV: tracking-event.v1 rows
    MV-->>CH: events, agg_metric_daily_v2, visitor/product read models 적재

4.2 분석 조회

sequenceDiagram

5. 실제 연동 방법

고객사가 https://www.priviatravel.com/이라고 가정하면, 무단 크롤링 방식이 아니라 고객사 웹/앱/백엔드에서 발생한 합법적 1st-party 이벤트를 Sunrise 수집 API로 전송해야 합니다. 개인정보 고지, 동의, 보존 정책, PII 비식별 처리는 고객사 운영 정책에 포함되어야 합니다.

5.1 로컬 compose 실행

docker compose up -d --build
docker compose ps

현재 로컬 compose는 .env의 값을 사용합니다. 기본 API key는 test-prod-key이고 tenant는 tenant-a입니다.

5.2 ClickHouse 물리 DB 확인

bash docs/api/clickhouse_smoke.sh
docker compose exec -T clickhouse clickhouse-client \
  --user sunrise \
  --password test-clickhouse-password \
  --database sunrise \
  --query "SELECT name, engine FROM system.tables WHERE database = currentDatabase() ORDER BY name"

응답이 eligible=true이면 SDK는 creative와 items를 사용해 팝업/배너를 노출하고, 이어서 /v1/onsite/impressions, /v1/onsite/clicks, /v1/onsite/dismissals로 성과 이벤트를 보냅니다.

5.9 Postman 사용

docs/api/sunrise.postman_collection.json을 import한 뒤 컬렉션 변수 sunrise_api_key를 실제 API key로 설정합니다. 기존 Postman Environment에 api_key가 있어도 영향을 받지 않도록 컬렉션 변수명을 분리했습니다.

  1. base_url: 로컬은 http://127.0.0.1:8000, 운영은 API 도메인
  2. sunrise_api_key: 로컬은 test-prod-key, 운영은 고객사 key
  3. Ingestion > Collect Seed Events 실행
  4. Ingestion > Wait for Kafka to ClickHouse 실행
  5. Audience > Templates 또는 Cart Templates 실행
  6. Recommendation > Upsert Product Value Features 실행
  7. Onsite > Decide Campaign 및 tracking 요청 실행
  8. Analytics/Prediction/Recommendation/AI 요청 실행

5.10 sunrise-core API key 발급 예시

curl -X POST "http://localhost:8080/api/api-keys" \
  -H "Content-Type: application/json" \
  -H "X-Sunrise-Tenant: tenant-a" \
  -H "X-Sunrise-Actor-Id: admin-1" \
  -H "X-Sunrise-Actor-Type: USER" \
  -d '{
    "name": "Privia production ingest key",
    "scopes": ["INGEST", "READ_ANALYTICS"],
    "expiresAt": "2027-06-03T00:00:00Z"
  }'

sunrise-core가 반환하는 secret은 최초 1회만 노출되는 값으로 다뤄야 합니다. 현재 sunrise는 환경변수 JSON 기반 API key 매핑을 사용하므로, 실제 운영에서는 core의 key 저장소와 sunrise 인증 계층을 Secret Manager/IAM/DB 조회 방식으로 연결하는 후속 작업이 필요합니다.

6. 운영 체크리스트

영역 현재 구현 운영 시 확인
인증 X-Sunrise-Key → tenant_id 매핑, core는 API key hash/prefix 저장 정적 env JSON 대신 Secret Manager 또는 core DB 연동, rotation/revoke 반영
수집 안정성 payload size limit, rate limit, Kafka retry, DLQ topic API gateway 제한, tenant별 quota, DLQ 알림/재처리 운영
OLAP ClickHouse Kafka Engine, Materialized View, daily read model, product_features_v1 replication, retention, partition TTL, feature schema migration 정책
ML JSON artifact 기반 prediction scoring, recommendation_ranker v3 logistic ranking 모델 registry, offline training data pipeline, A/B 검증, drift monitoring
추천 상품 가치 /v1/recommendations/products로 가격/할인/평점/리뷰/반품률/마진/재고 upsert 상품 카탈로그/PIM/예약 재고 시스템과 자동 동기화, 품질 feature freshness SLA
온사이트 실행 /v1/onsite/decide, impression/click/dismiss 수집 API 활성 캠페인 저장소, frequency cap 저장소, 실험군 배정, 프론트 SDK 연동
오디언스 /v1/audiences/templates로 50+ 기본 템플릿 catalog 제공 조건 DSL 편집기, 실시간 모수 preview, snapshot materialization 연동
개인정보 visitor_id 문자열만 계약상 요구 PII 직접 저장 금지, consent, 삭제 요청, 보존 기간 정책
관측성 /ops/metrics ingestion counters Prometheus/OpenTelemetry, Kafka lag, ClickHouse insert/query 알림
