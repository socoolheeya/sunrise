1. Analytics 기능과 역할

Analytics는 수집된 행동 이벤트를 마케터·콘솔이 읽는 분석 지표/리포트로 가공하는 읽기 전용(CQRS Read) 영역입니다. 쓰기(수집)와 분리되어, 무거운 집계는 목적별 read model로 미리 적재하고 조회는 캐시로 가속합니다.

기능역할(무엇을 답하나)핵심 산출

5. 용어집 (read model 외)

Read Model (읽기 모델)

쓰기용 원천(이벤트)과 분리해 특정 조회에 최적화된 형태로 미리 만들어 둔 데이터입니다. 매 조회마다 원천을 무겁게 집계하는 대신, refresh로 한 번 계산해 전용 테이블(order_facts·customer_segment_daily·cohort_retention)에 적재하고 조회는 그 테이블만 읽습니다. 정확성(중복제거 로직 1곳)·성능·재현성을 얻습니다.

Materialize / Refresh (머티리얼라이즈 / 갱신)

원천 이벤트로 read model을 계산해 테이블에 적재하는 작업. POST /…/refresh 엔드포인트(또는 스케줄러/cron)가 트리거합니다. SQL은 upsert/delete-then-insert, ClickHouse는 ALTER DELETE + insert로 멱등하게 갱신합니다.

CQRS (Command Query Responsibility Segregation)

쓰기(Command, 수집)와 읽기(Query, 분석)의 책임/모델을 분리하는 패턴. analytics는 Query 쪽으로, 읽기에 맞춘 read model과 캐시를 사용합니다.

주문 단위 중복제거 (order_id dedup)

같은 주문이 여러 purchase 이벤트(재시도·분할발송·스크립트 중복 발화)를 낼 수 있으므로, 매출·주문수는 order_id 기준으로 1건만 셉니다. order_facts(또는 raw 집계의 dict 키)로 보장합니다.

onsite_matched (온사이트 추적 매칭)

주문이 온사이트 스크립트로 추적된 주문인지 여부. 구현상 구매 이벤트에 session_id가 동반되면 true. 숨은 매출 산출의 기준이 됩니다.

숨은 매출 (hidden revenue)

총 매출 − 온사이트 추적 매출 = max(0, total − onsite). 스크립트가 추적하지 못한(추적 손실) 매출 규모를 나타냅니다. onsite_coverage_rate = onsite / total로 추적 커버리지를 봅니다.

기여 매출 · Attribution window · Last-touch

기여 매출(attributed): 캠페인 노출/클릭 이후 일정 시간(attribution window) 안에 발생한 구매. last-touch: 구매 직전 window 내 가장 마지막 접점 채널에 매출을 귀속하는 규칙(model=last_touch).

코호트 · 잔존율(Retention) · cohort_type · granularity · offset

코호트: 같은 시점에 첫 활동을 한 고객 집단(코호트 라벨=첫 활동 버킷). granularity: 버킷 단위(day/week/month). offset: 코호트 기준 경과 버킷 수(0=첫 버킷). retention_rate=해당 offset 잔존 인원/코호트 모수. cohort_type: 코호트 정의 이벤트(visit=방문류, purchase=구매).

Lifecycle 세그먼트 (visit / purchase segment)

고객을 최근성 기준으로 분류. visit_segment: 마지막 방문 경과일 ≤7 active / ≤30 risk / 그 외 inactive. purchase_segment: 구매 0 no_purchase / 마지막 구매 ≤30 active / ≤90 risk / 그 외 inactive.

세그먼트 이동 (transition) · as_of · "absent"

두 시점(as_of=스냅샷 기준 시각) 스냅샷을 비교해 고객이 어느 세그먼트→어느 세그먼트로 옮겼는지 집계. 이전 스냅샷에 없던 신규 고객의 출발 세그먼트는 "absent". transition_rate=(from→to 인원)/(이전이 from인 현재 고객 총원).

Benchmark · delta_ratio

테넌트 지표를 플랫폼(전체 테넌트) 평균과 비교. delta_ratio = (tenant − benchmark) / benchmark (cvr·aov·repeat_rate).

유입(Inflow) · 채널(channel)

방문/구매가 어디서 왔는지. 채널은 utm_medium ?? utm_source ?? category ?? "unknown" 우선순위로 결정하고, 채널별 세션·구매·매출·CVR·AOV를 집계합니다.

DataTalk · Frozen Snapshot

DataTalk: 일일 사이트 프로파일링 리포트(지표·퍼널·매출·top 유입·이상치). frozen snapshot: 그 리포트를 특정 시점에 고정(불변)해 저장한 것 — 나중에도 같은 값을 재현. status=frozen.

이상치(Anomaly)

DataTalk가 표시하는 단순 규칙 신호: no_sessions(방문 0), no_conversion(방문은 있으나 전환 0), low_onsite_tracking_coverage(매출 있으나 온사이트 커버리지 < 0.5).

테넌트 격리 (Tenant Isolation / Multi-tenancy)

여러 쇼핑몰(테넌트)이 한 시스템을 공유하므로 모든 쿼리에 tenant_id를 강제해 데이터가 섞이지 않게 합니다. 예외는 벤치마크용 플랫폼 평균(전체 테넌트 익명 집계)뿐.

일 경계 캐시 · 캐시 키

조회 윈도우를 일(day) 경계로 정렬(end=내일 0시 UTC)해 하루 동안 캐시 키가 안정되도록 합니다. 키={schema}:{tenant}:{resource}:{start}:{end}. refresh 시 관련 키를 무효화(delete/prefix)합니다.

MetricInputs / DashboardMetrics

MetricInputs: 원시 집계값(방문자·구매자·주문수·매출·재구매자·세션). DashboardMetrics: 거기서 파생한 지표 — CVR=구매자/방문자, AOV=매출/주문수, repeat_rate=재구매자/구매자.

ClickHouse 엔진 용어

ReplacingMergeTree(ver): 같은 ORDER BY 키의 행을 ver(예 received_at·computed_at) 최신본으로 collapse → 멱등 적재/중복 제거에 적합. AggregatingMergeTree: uniqState/sumState 같은 부분집계 상태를 병합 저장. Materialized View(MV): 원천 insert 시 자동으로 대상 테이블에 변환 적재하는 뷰. Kafka engine: Kafka 토픽을 테이블처럼 소비. argMax(x, t): t가 최대인 행의 x.

Port / Adapter / Clean Architecture

Port: 도메인이 정의한 인터페이스(AnalyticsRepository ABC). Adapter: 그 구현(SqlAnalyticsRepository·ClickHouseAnalyticsRepository). 유스케이스는 Port에만 의존하므로 백엔드 교체가 계약을 깨지 않습니다(domain ← application ← adapters).

멱등성(Idempotency) · event_id

같은 요청/이벤트가 여러 번 처리돼도 결과가 한 번 처리한 것과 같음. (tenant_id, event_id) 유니크 제약으로 중복 이벤트를 1건으로 보장하고, read model 적재(upsert/replace)도 멱등합니다.
