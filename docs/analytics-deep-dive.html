Analytics 모듈 상세 (코드 · UML · 시퀀스)

app/analytics/의 도메인·포트·어댑터(SQL/ClickHouse)·유스케이스·HTTP 라우터를 코드 수준으로 설명합니다. 모든 공식·흐름은 현재 구현 기준입니다.

개요 · 책임

Analytics는 읽기 전용(CQRS Read) 분석을 담당합니다. 수집된 행동 이벤트를 KPI·퍼널·코호트·벤치마크·유입·기여·숨은매출·세그먼트·DataTalk 리포트로 가공합니다. 무거운 집계는 목적별 read model로 머티리얼라이즈하고, 읽기는 Redis 캐시로 가속합니다.

applicationqueries.py유스케이스(Get*/Refresh*) — 포트에만 의존
adaptershttp.py · repository.py · clickhouse.pyHTTP 라우터 + SQL/ClickHouse Repository 구현

계층 구조 (의존성 방향)

domain ← application ← adapters. 도메인은 프레임워크/DB를 모릅니다. 유스케이스는 AnalyticsRepository 포트에만 의존하므로, SQL/ClickHouse 어댑터를 교체해도 유스케이스·HTTP 계약은 불변입니다.

Read Model 머티리얼라이즈 패턴 read model

order_fact·segment·cohort는 매 조회 raw 집계 대신, refresh가 도메인 순수함수로 산출해 테이블에 적재하고 조회는 테이블만 SELECT합니다. SQL과 ClickHouse가 동일 의미로 동작합니다(대칭).

metrics — 핵심 지표 GetDashboardMetrics

metric_inputs로 원시 집계 → DashboardMetrics.from_inputs로 비율 계산. 구매는 order_id 단위 dedup.

# repository.metric_inputs: 이벤트 스캔 후 집계 visitors = set(visitor_id) sessions = set(session_id) purchase_orders = { order_id : (visitor, amount) } # order_id 첫건만 채택 repeat_purchaser = |{ visitor : 주문수 ≥ 2 }| # DashboardMetrics.from_inputs (model.py) cvr = purchaser_count / visitor_count # visitor=0 → 0 aov = revenue / purchase_count # purchase=0 → 0 repeat_rate = repeat_purchaser_count / purchaser_count session_count = sessions or visitor_count

GET /v1/analytics/metrics

funnel — 퍼널 GetFunnel

funnel_visitor_counts(타입별 distinct visitor) → 단계 구성 + 이탈률.

단계 = 조회(view) → 장바구니(cart_add) → 구매(purchase) drop_off(i) = (visitors[i−1] − visitors[i]) / visitors[i−1] overall_conversion = visitors[last] / visitors[first]

GET /v1/analytics/funnel

order_fact · revenue breakdown read model

order_fact.py fold_order_facts가 구매를 주문 단위로 dedup하고 onsite_matched/attributed를 계산해 테이블에 적재. 조회는 테이블에서 합산.

sequenceDiagram participant Op as Operator participant API as /order-fact/refresh participant R as Repository participant T as order_facts Op->>API: POST refresh (window, attribution_window_h) API->>R: 이벤트(구매+캠페인) 조회 R->>R: fold_order_facts() (순수함수) R->>T: 적재(SQL upsert / CH ALTER DELETE+insert) API->>API: order-fact breakdown 캐시 무효화 Op->>API: GET /order-fact/revenue-breakdown API->>R: order_revenue_breakdown() R->>T: SELECT(완료 주문) → 합산
# fold_order_facts order_id 동일 purchase 다중 이벤트 → 첫 1건 채택(dedup) onsite_matched = (purchase 이벤트에 session_id 존재) attributed = ∃ campaign touch t: 0 ≤ (purchase_t − t) ≤ window (last-touch) # revenue_breakdown_from_facts (status=completed) total = Σ amount · onsite = Σ[onsite_matched] · attributed = Σ[attributed] hidden = max(0, total − onsite) # RevenueBreakdown.hidden_revenue onsite_coverage_rate = onsite / total
레거시 GET /revenue-breakdown(방문자-touch 근사, deprecated)와 정확 경로 /order-fact/revenue-breakdown 병존. total/hidden은 양쪽 order dedup 동일, onsite 정의만 다름.

POST /order-fact/refresh · GET /order-fact/revenue-breakdown
